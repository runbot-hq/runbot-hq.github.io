Frequently Asked Questions

Is it free? Is it open source? Why isn't it on the App Store?

Yes and yes. RunBot is completely self-contained — no Apple Developer account required. You can fork it, customize it, and run your own build.

The app is free because for us it pays for it self in saved Cloud CI/CD spend. It's not on the App Store because Apple tends to clamp down on apps that access files and Terminal — that may change in the future.

Besides cost and speed, what makes local runners better?

Quite a lot. Local runners can run headed UI tests (impossible on cloud). They can test against beta OS versions and the latest Xcode the day they drop. Caches persist between runs, so dependencies don't re-download every time. And since you own the machine, there's no queue — your run starts the moment you push. Additionally, running builds locally keeps your proprietary code within your environment, significantly increasing privacy and security.

How many runners can I run on one machine?

As many as you like — each runner is a lightweight process. A 16 GB M1 MacBook handles 4–6 comfortably. A newer MacBook Pro can push 10–12. A Mac mini M4 with 64 GB can run 16–20. Stack enough runners and most CI pipelines complete in seconds, not minutes.

Which AI agents work with the failure hook?

Any CLI tool that accepts terminal input works. The default uses Cline CLI, but you can swap in claude, codex, or anything else.

To configure it, go to the repo in Settings and edit the hook command. For example, to call Claude Code:
claude -p "your instructions here" --dangerously-skip-permissions

Why does RunBot need GitHub permissions?

It only asks for the minimum needed. repo unlocks workflow status, logs, and runner registration; read:org + admin:org lets it monitor org-level runners; workflow enables re-running or cancelling runs.

RunBot cannot read your source code, write to repos, or open issues, and your token never leaves your Mac's Keychain. Prefer to skip OAuth entirely? RunBot automatically picks up a token from the GH_TOKEN or GITHUB_TOKEN environment variable.

Does RunBot send any data anywhere?

No. RunBot only contacts api.github.com for the REST API and github.com for sign-in. There's no analytics, no telemetry, no crash reporting, and no third-party calls. All run data is held in memory only and discarded on quit.

Can I use RunBot without local runners?

Yes. It works perfectly as a standalone workflow monitor. You get a live tree view of all your workflow runs, jobs, and steps. The AI failure hook also works without local runners — it fires whenever a monitored run fails, regardless of where it executed.

Why is CI/CD so important in the agentic era?

Agents loop until the code builds, tests pass, and every check goes green. Cloud-based Actions weren't built for intensive, long-running loops. Because agentic coding is non-deterministic, your CI/CD pipeline acts as a boundary layer (AI code reviewers, code-health gates like SonarQube, dependency scanners, UI tests) that turns wild agent output into something you can ship.

What technology stack powers RunBot?

RunBot is built entirely in Swift 6.2 using SwiftUI, with a modern actor-based architecture that keeps all UI state on the @MainActor and background work fully isolated in dedicated actors. Data flow is driven by Swift's native async/await and @Observable.

Get started

Up and running in minutes.

1

Install with one command

Run the command below in Terminal. RunBot installs and launches automatically.

curl -fsSL https://eoncode.github.io/run-bot/install.sh | bash
2

Sign in with GitHub

Automatically uses your CLI env GH_TOKEN / GITHUB_TOKEN variable. Or sign in with one click via OAuth.

3

Add a repo or org scope

Pick any repo or org that is available in your account. Monitor a single repo or an entire GitHub org.

4

Add & manage local runners (optional)

Provision new runners or pick up pre-existing ones. Start, stop, or remove them directly from the UI.

5

Configure a failure hook (optional)

Configure a shell command per repo or branch that sends a prompt to your AI CLI whenever a run fails.
